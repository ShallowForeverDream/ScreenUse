# GitHub 私有仓库同步

ScreenUse 使用独立的 GitHub 私有仓库保存端侧加密快照。公开源码仓库和个人数据仓库完全分离。

## 同步边界

同步内容：

- 分类、项目、任务；
- 工作会话及归因证据；
- 人工学习的归因规则；
- 删除墓碑和设备信息。

不同步内容：

- GitHub Token、同步密钥和其他设置密钥；
- 原始窗口采样、浏览器查询参数、日志、导出和数据库备份；
- 临时 AI 作业、计划导入缓存和本机诊断状态。

## 加密与凭据

快照先使用 gzip 压缩，再使用随机 nonce 的 AES-256-GCM 加密。远端文件只包含版本化加密信封、nonce 和密文。GitHub Token 与 256 位同步密钥通过系统凭据库保存，普通设置接口不会返回它们。

同步密钥不是 GitHub 密码。首台设备生成后，应通过可信渠道复制给其他设备；丢失后无法解密远端快照。更换密钥前应先保留本机数据库备份。

## 合并模型

每次同步执行：

1. 从当前 SQLite 生成本地快照；
2. 拉取远端文件并在本机解密；
3. 按实体 ID 和 `updated_at` 合并本地、远端记录；
4. 应用删除墓碑，删除时间晚于记录更新时间时以删除为准；
5. 在事务中写回本机数据库；
6. 加密合并后的快照，携带远端 SHA 更新 GitHub；
7. 若 SHA 已变化，重新拉取并合并后重试一次。

界面中的“拉取并推送”就是上述完整流程，而不是直接用远端 SQLite 覆盖本机数据库。这样 Windows、后续 Android 和平板端可以同时产生记录，再按实体 ID 合并，不会因为设备同时在线而损坏数据库。

项目、任务等父子实体按依赖顺序应用。设备列表只用于显示最近同步设备，不参与业务归因。

## 仓库规则

- 数据仓库必须是 Private；ScreenUse 会拒绝向 Public 仓库写入。
- 仓库不存在时可自动创建并初始化默认分支。
- 默认文件路径为 `screenuse/snapshot-v1.json.gz.enc`。
- 默认分支应与仓库默认分支一致。
- 不要把同步文件移动到 ScreenUse 的公开源码仓库。

## 恢复与排障

- 新设备先安装 ScreenUse，再填写相同仓库和同步密钥，执行一次手动同步。
- 每次远端更新都会创建 Git commit；“同步历史/撤销”会打开目标数据文件的提交历史，可恢复上一版且保留完整追溯链。
- 同步前可在“数据管理”创建 SQLite 备份；同步写入本地时使用事务。
- `401/403` 通常表示 Token 无效或无目标仓库权限。
- “仓库必须为 Private”表示目标仓库当前公开，应先改为私有或使用新的私有仓库。
- “无法解密”通常表示同步密钥不一致；不要用新密钥覆盖仍需恢复的远端数据。
